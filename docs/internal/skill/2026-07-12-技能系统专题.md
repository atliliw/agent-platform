# 技能系统专题

> 专题:Agent 技能(Skill)系统现状审计、对标与改进路线
> 日期:2026-07-12
> 目的:盘点本平台 Skill 系统已实现的能力,对标 Anthropic Agent Skills(本系统参照的模型)与其它开源框架,找出差距,给出逐项、可落地的改进方案与优先级,作为后续演进的选型依据。

---

## 1. 设计模型

本平台的 Skill 系统明说参照 **Anthropic Agent Skills(SKILL.md)模型**,核心理念是**渐进式披露(progressive disclosure)**:

- skill 的 `Name + Description` 始终注入 system prompt(廉价、常驻),让 agent 知道"有哪些技能可用";
- skill 的完整 `Instructions` **按需加载**--agent 调用内置 `load_skill` 工具时才取回,避免一个 agent 挂很多 skill 时 prompt 被撑爆;
- 挂载是 **many-to-many**:`Agent.Skills []string` 持有 skill ID,一个 skill 可被任意多 agent 挂载;skill 独立存于自己的 store,按 ID 引用。

### 1.1 数据模型

```
Skill (pkg/agent/skill.go)
├── ID, Name, Description      // Name+Description 常驻 prompt
├── Instructions               // 完整指令,load_skill 按需返回
├── Tools []string             // skill 声明依赖的工具;挂载时授予 agent(2026-07-12 接入)
├── Tags, Status(draft/active) // draft 不注入、不可 load
├── Version                    // 更新自增(目前装饰性,无锁定)
└── CreatedAt, UpdatedAt

Agent.Skills []string          // 挂载的 skill ID(proto/agent/agent.proto: skills=14)
```

### 1.2 核心机制(engine.go)

| 机制 | 位置 | 行为 |
|------|------|------|
| prompt 注入 Name+Description | `buildAgentMessages` ~L946 | 仅 active skill 的 Name+Description 进 prompt |
| load_skill 工具注册 | `buildAgentTools` ~L1107 | agent 有挂载 skill 时加入工具列表 |
| load_skill 执行 | `executeLoadSkill` ~L1142 | 在 agent.Skills 范围内按 name/ID 解析,返回完整 Instructions(安全边界:只能 load 已挂载的) |
| **工具动态门控** | `buildAgentTools` ~L1085 | 合并 active skill 的 `Tools` 进 agent 工具集,去重,只加 registry 已存在的工具(2026-07-12 新增) |
| 降级 | 各处 | skillStore 为 nil / 无挂载 / ID 解析失败 = 不注入、不报错 |

## 3. 对标

### 3.1 vs Anthropic Agent Skills(参照模型)

| 维度 | Anthropic | 本平台 | 差距 |
|------|-----------|--------|------|
| Name+Description 常驻 | ✅ | ✅ | 无 |
| Instructions 按需加载 | ✅ 模型读 SKILL.md | ✅ load_skill 工具 | 无 |
| 渐进式披露(核心) | ✅ | ✅ | **无--核心对标正确** |
| skill 捆绑可执行脚本/资源文件 | ✅ 目录带 .py/.sh | ❌ Instructions 纯文本;Tools 只能引已有工具 | **大** |
| 按请求自动发现相关 skill | ✅ 模型从整库自动选 | ❌ 手动挂载 + 显式 load | **中** |
| skill = 目录(可含多文件) | ✅ | ❌ 单条 DB 记录 | **大**(上一条的基础) |
| allowed-tools 语义 | 限制(收窄) | 授予(放大) | 语义相反 |
| skill 嵌套/子 skill | ✅ | ❌ 扁平 | 小 |
| 版本锁定 | 弱 | 弱(version 装饰性) | 持平 |
| 校验/状态机 | ✅ | ✅ draft/active + Validate | 无 |

### 3.2 vs 其它框架(定位差异)

- **Superpowers / Agent-Skills**(见 `docs/research/superpowers-vs-agent-skills.md`):是 **Claude Code CLI 插件层**的 skill(Shell 脚本、斜杠命令、流程强制),跑在开发者本机的 Claude Code 里。本平台 skill 是**运行时挂载到平台 agent 的能力模块**,跑在后端 engine 里。**不同层,不直接可比**,但它们的"skill = 流程性知识 + 可执行脚本"理念印证了下方 Gap 1 的重要性。
- **Semantic Kernel**:"plugin" 含 semantic function(prompt 模板)+ native function(代码),按 description 自动调用。对标点:自动调用(对应 Gap 2)+ skill 带代码(对应 Gap 1)。
- **LangChain / CrewAI**:以 tool 为一等公民,"skill"非核心抽象;prompt 模板与 tool 分离。本平台把"prompt 流程 + 工具门控"合进 Skill,比它们更接近 Anthropic 模型。

## 4. 差距与影响

### Gap 1:skill 不能捆绑代码/资源(最大)

- **现状**:Skill.Instructions 是纯文本;Skill.Tools 只能引用 registry 里**已存在**的工具。
- **后果**:没法做一个"带自定义逻辑"的 skill。例如解析私有格式、带特殊鉴权调某 API、跑一段特定算法--必须先把工具加进全局 registry,再让 skill 引用。Anthropic 的 skill 能自带 `.py` 脚本让模型直接跑,本平台不能。
- **本质**:Anthropic skill = 文本 + 代码;本平台 skill = 只有文本。这从根本上限制了 skill 能**做**什么。

### Gap 2:没有自动发现(中)

- **现状**:管理员必须**预先**把 skill 挂到每个 agent(`agent.Skills`);agent 只知道被分配的那几个。load_skill 也只能 load 已挂载的。
- **后果**:不能搞"100 个 skill 的库,agent 按上下文自己挑"。每个 agent 的技能视野被手动挂载锁死,扩展性和灵活性受限。Anthropic 是模型从整库按请求自动选。

### Gap 3:skill 是 DB 记录不是目录(支撑 Gap 1)

- **现状**:skill 存为 Mongo 单文档,Instructions 是一个文本字段。没法附加文件。
- **后果**:Gap 1 的基础。没有目录/资产结构,就无法捆绑脚本。

### Gap 4:allowed-tools 语义相反(低)

- **现状**:`Skill.Tools` 是**授予**(挂载即放大 agent 能力)。Anthropic 的 `allowed-tools` 是**限制**(skill 活跃时收窄 agent 可用工具,安全边界)。
- **后果**:不算错(授予也有用),但缺少"skill 激活时收窄工具防越权"的能力。与参照模型方向相反。

### 小项

- **version 装饰性**:更新自增,但挂载 by ID 永远用最新版,无版本锁定/回滚。
- **删 skill 不清悬挂引用**:挂载的 skill 被删后,`agent.Skills` 里残留 ID,运行时静默跳过(不崩),但不自动清理。
- **marketplace/导入导出**:skill 无法打包分享(推迟项)。

## 5. 改进方案(逐项)

每项给出:目标 / 方案(数据模型 + 代码改动)/ 工作量 / 优先级 / 风险。

### P0a — 删除 skill 时清理悬挂引用

- **目标**:删 skill 后,所有 agent 的 `agent.Skills` 自动移除该 ID,不留垃圾。
- **方案**:在 `agent_service.go` 的 `DeleteSkill` 里,遍历 agents,把该 ID 从 `Skills` 去掉并 upsert;或加一条后台清理。也可在 `MongoSkillStore` 层用 mongo 的 `$pull` 批量更新 `agents.skills`。
- **代码**:`services/agent-service/internal/service/agent_service.go` DeleteSkill;`pkg/agent/mongo_store.go` 加 `PullSkillFromAllAgents(skillID)`。
- **工作量**:~半天 | **优先级**:P0(便宜、防数据腐化)| **风险**:低。

### P0b — skill 导入/导出(marketplace 雏形)

- **目标**:skill 能导出为 YAML/JSON 文件、从文件导入,便于跨环境分享。
- **方案**:加 `GET /api/v2/skills/:id/export`(返回单文件 YAML)和 `POST /api/v2/skills/import`(接受 YAML/JSON,upsert)。前端技能库页加"导出/导入"按钮。
- **代码**:`skill_handler.go` 两个 handler;`frontend/src/pages/Skills/` 加按钮。
- **工作量**:~1 天 | **优先级**:P0 | **风险**:低(注意导入时 name 冲突策略:覆盖/拒绝/重命名)。

### P1 — 自动发现(AutoDiscoverSkills)

- **目标**:agent 可标记为"自动发现",prompt 里列出**全部 active skill** 的 Name+Description,load_skill 能 load 任意 active skill,不必预先挂载。对标 Anthropic 的整库自动选。
- **数据模型**:`Agent.AutoDiscoverSkills bool`(proto 加 field 15)。
- **代码**:
  - `engine.go buildAgentMessages`:若 `agent.AutoDiscoverSkills`,从 skillStore 取**全部 active** skill 的 Name+Description 注入(而非仅 `agent.Skills`)。
  - `executeLoadSkill`:auto-discover agent 允许 load 任意 active skill(放宽安全边界,或加独立路径 `executeLoadSkillAny`)。
  - proto + frontend agent 编辑器加开关。
- **权衡**:prompt 随 skill 数增长(每 skill 一行)。50 个以内可接受;>50 需加** embedding 检索**:用 skill description 做向量,按当前请求召回 top-K 注入(类似 memory recall)。
- **工作量**:基础版 ~2-3 天;加 embedding 检索再 +2-3 天 | **优先级**:P1 | **风险**:中(安全边界放宽需评审;prompt 体积需监控)。

### P2 — skill 捆绑脚本/资产(目录化)

- **目标**:skill 能自带脚本/资源文件,instructions 可指示模型运行捆绑脚本。对标 Anthropic 的 skill = 目录 + 代码。
- **数据模型**:skill 加 `Assets []SkillAsset{ Name, ContentType, Size, StoreRef }`,资产存 GridFS/对象存储(不入 Mongo 文档,避免大字段)。
- **新工具**:`run_skill_script(skill_id, script_name, args)`--取出捆绑脚本,**沙箱内**执行,返回 stdout/stderr。
- **沙箱**(难点):容器/gvisor/WASM 隔离;超时、内存、文件系统、出网白名单。这是本项工作量大头。
- **代码**:`skill.go` 加 Asset;新 `SkillAssetStore`;engine 加 `run_skill_script` 内置工具(同 load_skill 一样本地拦截,不走 MCP);前端技能编辑器加资产上传。
- **工作量**:~1-2 周(沙箱占大头)| **优先级**:P2(按需,有"skill 带自定义代码"诉求才做)| **风险**:高(沙箱安全是安全审查重点)。
- **替代轻量方案**:若不想做通用沙箱,可让 skill 只能携带**单个声明式脚本**,且只允许调用白名单解释器(python3),资源配额严格--降低但不下架风险。

### P3a — 版本锁定

- **目标**:挂载可锁定到具体版本,skill 更新不自动影响已锁定 agent。
- **方案**:挂载引用改为 `skill-id@version` 或独立 `Mount{ SkillID, Version }` 记录;load_skill/工具门控按锁定版本取。
- **工作量**:~2 天 | **优先级**:P3 | **风险**:中(需迁移现有 by-ID 挂载)。

### P3b — allowed-tools 限制语义

- **目标**(可选):skill 可声明 `AllowedTools`,加载该 skill 时把 agent 工具集**交集**收窄,防止 skill 越权用其它工具。
- **方案**:`Skill.AllowedTools []string`;skill 处于"活跃已加载"状态时,agent 有效工具 = `agent.Tools ∪ skill.Tools` 再 ∩ `skill.AllowedTools`。需要跟踪"当前活跃 skill"状态。
- **工作量**:~2 天 | **优先级**:P3(有安全收窄诉求才做)| **风险**:中(状态跟踪复杂度)。

## 6. 演进路线

| 阶段 | 项 | 做什么 | 为什么先做 |
|------|-----|--------|-----------|
| **现在(P0)** | P0a + P0b | 悬挂引用清理 + 导入导出 | 便宜、防数据腐化、立刻有用,不碰核心 |
| **短期(P1)** | 自动发现 | AutoDiscoverSkills + (可选)embedding 检索 | 中等工作量,把 skill 从"手动挂载"升级到"整库可用",UX 质变 |
| **中期(P2)** | 捆绑脚本 | 目录化 skill + 沙箱 run_skill_script | 大工程,只有"skill 需带自定义代码"的真实诉求才启动;启动后 skill 能力质变 |
| **按需(P3)** | 版本锁定 / allowed-tools 限制 | 按场景 | 多租户/强安全场景才需要 |

**判断锚点**:
- 若目标只是"可复用提示流程 + 工具门控"的 skill 系统--**现状已完整**,做 P0 即可收尾。
- 若要对标 Anthropic"整库自动选"--做到 P1。
- 若要对标 Anthropic"skill 带代码"--必须做到 P2。

---

## 7. 结论

1. **核心对标正确**:渐进式披露(Name+Description 常驻 + load_skill 按需 + 工具动态门控)实现完整、与 Anthropic 一致、已端到端验证。这是最难的部分,做对了。
2. **不算"完美"**:对标 Anthropic 还有三个实质差距--捆绑脚本(大)、自动发现(中)、目录化(支撑项)。这些决定 skill 能"做"什么、能"看到"什么。
3. **改进路径清晰**:P0(便宜收尾)→ P1(自动发现,UX 质变)→ P2(捆绑脚本,能力质变)。是否往下走,取决于平台对 skill 的定位:提示流程型已够用;Anthropic 级需 P1/P2。
4. **最近一次实质改进**:2026-07-12 接入工具动态门控(`buildAgentTools` 合并 `skill.Tools`),补上了"挂 skill 自动带工具"的直觉能力,并修正了 `skill.go` 的"documentation only"过期注释。
