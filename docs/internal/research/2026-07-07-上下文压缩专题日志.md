# 上下文压缩专题日志

> 专题:LLM Agent 上下文压缩(Context Compression)模式梳理
> 日期:2026-07-07
> 目的:把生产里真正在用的压缩模式系统化,映射到本项目的现状与演进路线,作为后续实现的选型依据。

---

## 1. 为什么需要上下文压缩

LLM Agent 每一轮决策都要把**完整对话历史**塞进 prompt 发给模型。随着对话变长、工具调用变多,prompt 会越滚越大,带来三个问题:

| 问题 | 表现 | 后果 |
|------|------|------|
| **token 浪费** | 老消息对当前决策无用,却每轮重发 | 成本随轮次线性甚至超线性增长 |
| **上下文窗口溢出** | 累积 token 超过模型上限 | 请求直接报错,对话中断 |
| **指令衰减** | system prompt 被海量历史淹没 | 模型偏离任务目标,质量下降 |

> 核心矛盾:模型上下文窗口有限(8K~1M token),而有效工作记忆需求无限。压缩的本质是**在有限窗口里放进「对当前决策最有用」的信息**。

### 1.1 两个关键区分

- **有损 vs lossless**:lossless 只截断/丢弃冗余,信息可恢复;有损(摘要)会把多条压成一条,细节不可逆。
- **减体积 vs 省成本**:减体积是让 prompt 变小以塞进窗口;省成本是让重复前缀不重算(prompt 缓存)。二者独立,常被混淆。

## 2. 九种主流压缩模式

### 模式 1:滑动窗口(Sliding Window)

**原理**:只保留最近 N 条消息,更早的直接丢弃。

```
[老消息...] [msg-N] ... [msg-1] [current]
              ↑ 只保留这 N 条
```

- **谁在用**:基础 ChatGPT、LangChain `ConversationBufferWindowMemory`、多数简单 chatbot
- **优点**:实现最简单、无额外 LLM 调用、完全确定性
- **缺点**:老上下文全丢,边界处信息突变
- **适用**:短对话、对历史不敏感的场景

### 模式 2:按 token 预算截断(Token-Budget Truncation)

**原理**:system prompt 全留 + 最近 N 条压在 token 预算内,中间部分丢弃。比模式 1 多了「按字数/token 而非条数」的约束。

```
[system 全留] [中间丢弃...] [最近 N 条,总 token ≤ 预算] [current]
```

- **谁在用**:多数生产 agent、Claude Code 单条消息预算控制
- **优点**:成本有明确上限、可预测
- **缺点**:边界处仍丢信息;需要 token 估算逻辑
- **本项目现状**:`MaxHistoryLength` 按**条数**截,属于模式 1 的弱版本,没有 size 预算 → **缺模式 2**

### 模式 3:摘要压缩(Summarization)

**原理**:定期用一次小模型调用,把老消息摘要成一条 compact 消息,最近若干条保留原文。

```
[summary: 老对话的要点] [recent N 条原文] [current]
   ↑ 多一次 LLM 调用生成
```

- **谁在用**:ChatGPT 压缩上下文、Claude Code `/compact`、LangChain `ConversationSummaryMemory`
- **优点**:保住要点、体积骤降、可处理极长历史
- **缺点**:多一次 LLM 调用(延迟+成本)、**有损**、摘要可能丢关键细节、摘要本身也会累积
- **适用**:长对话、需要长期连贯性的场景

### 模式 4:摘要 + 缓冲混合(Summary + Buffer)

**原理**:模式 2 + 模式 3 的组合。最近 N 条原文,更老的摘要,再老的可以递归摘要(分层摘要)。

```
[旧摘要(超老)] [较新摘要(老)] [最近 N 条原文] [current]
```

- **谁在用**:LangChain `ConversationSummaryBufferMemory`、成熟聊天产品
- **优点**:平衡最好,既有近期细节又有长期要点
- **缺点**:实现复杂,摘要层级的边界管理麻烦
- **适用**:产品级长对话系统

### 模式 5:检索式外部记忆(Retrieval-Based External Memory)

**原理**:历史**不入** context,而是 embed 后存到外部向量库;每轮按当前 query 检索 top-k 相关片段注入。

```
历史 → embed → 向量库
当前 query → 检索 top-k → 注入 context
```

- **谁在用**:mem0、MemGPT archival memory、**本项目 memory service**(layered memory)
- **优点**:理论无限历史、只注入相关项、跨会话可用
- **缺点**:检索可能漏召回、embedding 成本、检索延迟
- **本项目现状**:已实现(cases/episodic/semantic/working + checkpoint)→ **已有模式 5**

### 模式 6:工具结果压缩(Tool-Output Compression)

**原理**:专门压缩 tool 调用返回的结果(大 JSON、文件内容、搜索结果),因为它们通常占 prompt token 的 80% 以上,而原文往往不必要全留。

```python
# tool 返回 50KB JSON
result = call_tool(...)
# 压成要点
result = compress_tool_result(result)  # 截断 / 抽字段 / 摘要
```

- **谁在用**:Claude Code(工具结果截断)、多数 coding agent、Cursor
- **优点**:**性价比最高**,一处压缩省掉大量 token
- **缺点**:可能丢所需细节,需要按 tool 类型定制压缩策略
- **本项目现状**:`engine.go` 的 `buildAgentMessages` 把 `r.Result` **原样无上限**拼进 system prompt → **缺模式 6**

### 模式 7:LLM 自主分页(LLM-Managed Paging)

**原理**:借鉴操作系统虚拟内存。agent 自己调用函数(`memory_write` / `memory_recall`)把记忆在「主上下文」和「外部归档」之间 page in/out,由 LLM 决定什么重要。

- **谁在用**:MemGPT、Letta
- **优点**:自适应,agent 自己决定什么该留
- **缺点**:实现复杂、agent 可能管理失误、额外函数调用开销
- **适用**:前沿研究型 agent、长程自主任务

### 模式 8:重要性打分(Importance Scoring)

**原理**:给每条消息打分(recency × importance × relevance),驱逐低分消息。是模式 1/2 的智能化版本。

```python
score = α * recency + β * importance + γ * relevance_to_current_query
evict lowest-score messages
```

- **谁在用**:MemGPT(部分)、研究系统
- **优点**:驱逐策略更聪明,保留真正重要的而非仅最近的
- **缺点**:打分需要启发式或额外调用,调参难
- **适用**:对记忆质量要求高的系统

### 模式 9:Prompt 缓存(Prompt Caching)

**原理**:缓存稳定的前缀,重复部分不重新计算 KV cache。**不减体积,只省成本和延迟**。

```
[cached: system + tools + 稳定历史] [variable: 最近几条] 
         ↑ 命中缓存,0 计算成本
```

- **谁在用**:Anthropic prompt caching(5min/1hr TTL)、OpenAI 自动缓存
- **优点**:稳定前缀的边际成本趋近于零、延迟大降
- **缺点**:**不减小 prompt 体积**(窗口压力不变)、需要前缀真正稳定、缓存过期失效
- **适用**:几乎所有生产系统(作为省钱层叠加)

## 3. 真实系统的组合策略

生产里**几乎没有人只用一种**,典型组合:

| 系统类型 | 组合 | 说明 |
|---------|------|------|
| 基础 agent | `1 滑动窗口 + 6 工具结果压缩` | 最低门槛,止血用 |
| 长对话产品 | `1 + 6 + 3 摘要` | 对话一长自动触发摘要 |
| 跨会话系统 | `1 + 6 + 5 检索记忆` | 历史持久化,按需召回 |
| 自主长程 agent | `1 + 6 + 5 + 7 自主分页` | agent 自管理记忆 |
| 省钱层 | `上述任一 + 9 prompt 缓存` | 稳定前缀零成本复用 |

> 经验法则:**模式 6(工具结果压缩)永远是性价比最高的第一步**,因为 agent 场景下工具返回占 token 大头;**模式 9(prompt 缓存)是叠加在所有方案上的省钱层**;模式 3(摘要)是真正解决「长对话」的核心,但有损需谨慎。

## 4. 本项目现状映射

| 模式 | 现状 | 证据 |
|------|------|------|
| 1 滑动窗口 | ✅ 弱版本(按条数) | `MaxHistoryLength` count-based |
| 2 token 预算截断 | ❌ 缺失 | 无 size 上限 |
| 3 摘要压缩 | ❌ 缺失 | grep 无 summarization 逻辑 |
| 4 摘要+缓冲 | ❌ 缺失 | 依赖 3 |
| 5 检索式记忆 | ✅ 已实现 | layered memory + checkpoint |
| 6 工具结果压缩 | ❌ 缺失 | `r.Result` 无上限拼进 system prompt |
| 7 LLM 自主分页 | ❌ 缺失 | 未实现 |
| 8 重要性打分 | ❌ 缺失 | 未实现 |
| 9 prompt 缓存 | ❌ 未启用 | 客户端未配置 cache_control |

**结论**:本项目在「外部记忆」(模式 5)上已经做得不错,但「context 内部压缩」(模式 2/3/6)基本空白。当前最长板是检索记忆,最短板是 in-context 体积控制。

## 5. 演进路线建议

按**风险从低到高、收益从直接到边际**排序:

### v1 — 止血(lossless,零风险) ✅ 已实现(2026-07-07)
- **模式 2 + 模式 6**
- 分层 token 预算截断(system 12K / 最近 8 条各 6K / 更早各 1K)
- `AgentHistory.r.Result` 截到 ~1K 字
- 实现:`CompressingClient` decorator(镜像现有 `metricsClient` 模式)
- 收益:立刻卡住 token 上限,零行为变化
- 详见:`docs/plans/2026-07-07-context-compression.md`
- **实测**:部署 agent-service + chat-service 后,发送 7573 字符长消息,agent-service 日志:
  ```
  [compression] prompt truncated: ~2490 -> ~2100 tokens (saved ~390)
  [compression] prompt truncated: ~3814 -> ~3424 tokens (saved ~390)
  ```
  每次调用省 ~390 token(≈1573 字符 × 0.25 token/字符),与阈值数学吻合。正常短对话不触发压缩(低于阈值)。

### v1.1 — 省钱层
- **模式 9 prompt 缓存**
- 给稳定 system prompt 前缀打 `cache_control`
- 收益:边际成本趋零,不改体积
- 风险:低(仅成本优化)

### v2 — 真正的压缩
- **模式 3 摘要 + 模式 4 混合**
- 老消息定期摘要成一条,最近 N 条保留原文
- 收益:解决「长对话丢老上下文」
- 风险:中(有损、多一次调用、摘要质量需验证)

### v2+ — 智能化
- **模式 7 LLM 自主分页 / 模式 8 重要性打分**
- agent 自管理记忆,按重要性驱逐
- 收益边际递减,复杂度陡增
- 风险:高(需充分评估)

## 6. 选型决策树

```
要不要压缩?
├─ 否 → 至少上 模式 9 prompt 缓存(省钱)
└─ 是
   ├─ 工具返回占大头? → 先上 模式 6(性价比最高)
   ├─ 对话会变很长? → 加 模式 3 摘要
   ├─ 需要跨会话记忆? → 用 模式 5 检索(本项目已有)
   └─ 预算敏感? → 叠加 模式 9 缓存
```

---

## 7. 关键设计原则

1. **lossless 优先**:能截断解决就别摘要,有损方案最后再上
2. **不破坏 tool 配对**:压缩 tool_call / tool_response 时必须成对处理,否则模型会困惑
3. **配置驱动**:所有阈值走 config,可关可调,不写死
4. **可观测**:压缩前后 token 数要能度量(「省了多少」要看得见)
5. **decorator 模式**:压缩作为独立层插入,不侵入业务逻辑,可热插拔
6. **保留 system + 最近 N**:这两段永远原文保留,只压中间和工具结果
