如何实现 Multi-Agent 架构
让多个 AI 助手协作完成复杂任务
Multi-Agent 就是让多个专业 AI 助手协作完成任务。就像公司分工：老板分配任务、研究员搜索信息、程序员写代码、分析师做报告。
🎯 实现 Multi-Agent 的 4 个核心步骤
1️⃣
定义 Agent每个 Agent 有自己的"性格"（Instructions）、"能力"（Tools）、"协作关系"（Handoffs）
2️⃣
实现 HandoffAgent 之间通过 Handoff 交接任务，Engine 自动将 handoffs 转换为工具
3️⃣
实现 Engine核心循环：不断调用 LLM，问"下一步干啥"，根据响应决定继续还是结束
4️⃣
实现 ExecutionContext跨 Agent 共享状态：Variables、Messages、AgentHistory
💡 参考 OpenAI Swarm 设计思想

核心：Handoff 让 LLM 自己决定流程，简单任务直接处理，复杂任务分工协作
1
Agent 定义 + Handoff 机制
第一步：定义 Agent，第二步：实现交接
Agent 三要素
1

📝 Instructions（性格）

System Prompt，定义 Agent 的行为方式。比如："你是研究助手，搜索前先分析关键词，优先使用知识库..."

2

🔧 Tools（能力）

能调用的工具列表。Researcher 有 web_search、knowledge_search；Coder 有 code_execute、file_write

3

🤝 Handoffs（协作关系）

能交接给谁。Main Agent 的 handoffs: ["researcher", "coder"]，表示可交接给这两个人

// Agent 定义示例 { "id": "main-agent", "instructions": "你是主调度助手，分析请求分配给专业Agent", "tools": [], // 自己没有工具 "handoffs": ["researcher-agent", "coder-agent"] }
Handoff 工作原理
Handoff = 特殊的工具，但它和普通工具不同：普通工具执行返回结果，Handoff 会切换 Agent！
1

声明 Handoffs

Agent 配置中声明 handoffs: ["coder-agent"]

↓ Engine 自动注入
2

注入工具

Engine 自动创建 transfer_to_coder-agent 工具，和普通工具一起传给 LLM

↓ LLM 决策
3

切换 Agent

LLM 调用 transfer_to_coder-agent → Engine 检测到 → 切换到 coder-agent

// handoffs: ["coder-agent"] 会自动生成这个工具： { "name": "transfer_to_coder-agent", // 特殊前缀! "description": "Transfer to Coder Agent", "parameters": {} // 无参数 }
🔑 关键理解：LLM 不区分 Handoff 和普通工具！
对 LLM 来说都是工具，Engine 通过 transfer_to_ 前缀识别
2
Engine 执行循环
Multi-Agent 的心脏，不断问 LLM "下一步干啥"
Engine 是核心循环，每次循环都调用 LLM，根据响应决定继续还是结束。每次循环 = 1 次 LLM 调用！
循环的 5 个步骤
1

选择 Agent

入口 Agent 或 Handoff 后的新 Agent。execCtx.CurrentAgent 记录当前是哪个 Agent。

2

构建 Prompt

System Prompt + 上下文变量（Variables） + 执行历史（AgentHistory） + 对话历史（Messages）

3

构建 Tools

Agent 自己的工具 + 自动注入 Handoff 工具（buildTools 函数负责）

4

调用 LLM

llmClient.Chat(messages, tools)。发送消息和工具，等待 LLM 思考并返回。

5

处理响应

无工具调用 → 返回结果，结束。有工具调用 → Handoff 切换 Agent，普通工具执行后继续循环

for step := 0; step < MaxSteps; step++ { // 最多10步 // 构建 Prompt 和 Tools messages = buildMessages(currentAgent, execCtx) tools = buildTools(currentAgent) // 含Handoff工具 // 调用 LLM response = llmClient.Chat(messages, tools) // 检查是否有工具调用 if len(response.ToolCalls) == 0 { return response.Content // 无工具 → 结束! } // 处理工具调用 for tc in response.ToolCalls { if IsHandoffTool(tc.Name) { 切换Agent(), continue // Handoff → 切换 } else { 执行工具(), continue // 普通工具 → 执行 } } }
响应处理逻辑
1️⃣
无工具调用LLM 直接返回内容 → 任务完成，返回结果给用户，结束循环
2️⃣
Handoff 工具调用检测到 transfer_to_ 前缀 → 切换 Agent，继续下一轮循环
3️⃣
普通工具调用执行工具（如 web_search）→ 添加结果到 Messages → 继续循环
💡 每次循环 = 1 次 LLM 调用

这就是为什么 Multi-Agent 消耗 Token 多！
但也让 LLM 有机会在每一步做决策

ExecutionContext + 完整流程
跨 Agent 共享状态 + 实战示例
ExecutionContext 共享状态
多个 Agent 如何协作？靠ExecutionContext共享信息！这是跨 Agent 共享状态的容器，让新 Agent 能知道之前发生了什么。
📦 Variables 共享变量，传递结构化数据，如搜索结果
💬 Messages 完整对话历史，所有 Agent 的对话
📋 AgentHistory 执行路径追踪，记录每一步：哪个 Agent 做了什么
🔧 ToolResults 工具调用结果缓存
实际例子：AgentHistory 如何让 Agent 知道彼此干了什么
用户："研究 Go 并发，写代码"
Step 1: Researcher 搜索 → 结果存入 Variables 和 Messages
Step 2: Researcher Handoff → AgentHistory 记录交接
Step 3: Coder 开始时，buildMessages 会添加到 System Prompt：
"之前 Researcher 搜索了 Go 并发" + "搜索结果: Go 并发基于 CSP..."
结果：Coder 就知道 Researcher 找到了什么！
完整执行流程示例
用户："研究 Go 并发，写代码"
循环 1: Main Agent │ LLM 思考: "这是研究+编程任务，先研究" │ Tool Call: transfer_to_researcher-agent │ Engine: 检测 Handoff → 切换 Agent ↓ 循环 2: Researcher Agent │ LLM 思考: "需要搜索 Go 并发相关信息" │ Tool Call: web_search("Go concurrency") │ Engine: 执行工具 → 结果: "找到文章..." │ 添加结果到 Messages, AgentHistory ↓ 循环 3: Researcher Agent │ LLM 思考: "搜索完了，交给 Coder 写代码" │ Tool Call: transfer_to_coder-agent │ Engine: 检测 Handoff → 切换 Agent ↓ 循环 4: Coder Agent │ Prompt 包含: Researcher 的搜索结果 + AgentHistory │ LLM 思考: "根据搜索结果，写示例代码" │ Tool Call: code_execute │ Engine: 执行代码 → 成功 ↓ 循环 5: Coder Agent │ LLM 思考: "任务完成" │ 无 Tool Call → 直接返回内容 │ Engine: 结束! 返回结果 ✅
📊 统计：5 次循环 = 5 次 LLM 调用
消耗约 3000+ Tokens

每次循环 LLM 都能看到：上下文 + 历史 + 工具结果

为什么用 Handoff + 防止无限循环
对比分析 + 安全机制
为什么用 Handoff 而不是固定流程？

❌ 固定流程

必须按顺序执行：Research → Analyze → Code
用户说"直接写代码"，但必须先 Research
新增 Agent 要修改 Workflow 代码
不灵活，适应性差
VS

✅ Handoff

LLM 智能决策何时交接
用户说"直接写代码" → Main 直接 Handoff to Coder
新增 Agent 只需注册并声明 handoffs
动态流程，灵活适应不同任务
任务对比
任务 1："直接写一个 Hello World"
固定流程：Research(无用) → Code → 结果（浪费一步）
Handoff：Main → 直接 Coder → 结果（智能跳过 Research）
任务 2："研究 Go 并发，写代码"
固定流程：Research → Code（流程对了）
Handoff：Main → Researcher → Coder（LLM 决策）
防止无限循环的机制
1

MaxSteps = 10

最大步数限制。循环最多 10 次，超过强制停止，返回 "Maximum steps reached"

2

Handoff 验证

只能交接给 handoffs 列表里的 Agent。如果 Main 尝试交接给未声明的 Agent → ErrInvalidHandoff

3

AgentHistory 追踪

记录每一步，可检测循环模式：main → researcher → main → researcher（重复交接）

// Handoff 验证逻辑 func ValidateHandoff(fromAgent, toAgentID, registry) { // 1. 目标 Agent 存在? if !registry.Exists(toAgentID) { return ErrAgentNotFound } // 2. fromAgent 允许交接给 toAgentID? if !fromAgent.CanHandoffTo(toAgentID) { return ErrInvalidHandoff // 不允许! } return nil } // CanHandoffTo 实现 func (a *Agent) CanHandoffTo(id string) bool { for _, h := range a.Handoffs { if h == id { return true } } return false }
🛡️ 三重保护机制

步数限制 + 交接验证 + 历史追踪
确保不会无限循环或随意交接

Engine 循环 = 不停问 LLM "下一步干啥"
