# 01 - 项目概述

## 1. 项目背景

### 1.1 来源项目

本项目整合了两个现有项目的核心能力：

| 来源项目 | 技术栈 | 核心能力 |
|---------|--------|---------|
| langchainRustAgent | Rust + Axum | RAG 检索、A2A 协议、长期记忆、文档处理 |
| mcpgo | Go + Gin | Agent 对话、MCP 协议、Harness 运维治理系统 |

### 1.2 为什么新建而不是融合

- **技术栈统一**：使用 Go 作为统一语言，降低维护成本
- **架构升级**：从单体升级到微服务，支持独立部署和扩展
- **协议标准化**：使用 gRPC + Protobuf，便于跨语言调用
- **清晰的边界**：每个服务职责明确，避免代码耦合

## 2. 项目目标

### 2.1 核心目标

构建一个**生产级 AI Agent 运维平台**，具备：

1. **Agent 能力**
   - 单 Agent 工具调用
   - 多 Agent 协作（Master-Worker 模式）
   - 跨服务 Agent 通信（A2A 协议）

2. **知识库能力**
   - 文件上传（支持 PDF/Word/Markdown/TXT/JSON/CSV）
   - 智能分块（Token/Semantic/Markdown 策略）
   - 多模式检索（向量/BM25/混合）
   - 向量化存储（Qdrant）

3. **记忆能力**
   - 会话级短期记忆
   - 跨会话长期记忆（重要信息持久化）
   - 语义召回（基于向量相似度）

4. **运维治理能力**
   - 规则引擎（约束、权限、预算）
   - 输入输出护栏（内容过滤）
   - 评估测试（RAGAS + LLM-as-Judge）
   - A/B 测试（统计显著性）
   - SLO 管理（错误预算）
   - 混沌工程（故障注入）
   - 自动回滚
   - 根因分析
   - 成本分析
   - 特性开关
   - Pipeline 编排

### 2.2 非功能性目标

| 目标 | 要求 |
|------|------|
| 高可用 | 单服务故障不影响其他服务 |
| 可扩展 | 支持水平扩展，K8s 部署 |
| 低延迟 | P99 < 500ms（不含 LLM 调用） |
| 多租户 | 支持租户隔离、配额管理 |
| 可观测 | 完整的 tracing/metrics/logs |

## 3. 用户场景

### 3.1 场景一：企业知识库问答

```
用户上传公司文档 → 系统自动分块向量化 → 用户提问 → 
系统检索相关知识 → Agent 生成回答 → 存入长期记忆
```

### 3.2 场景二：多 Agent 协作任务

```
用户提交复杂任务 → Planner Agent 分解任务 → 
分配给 Specialist Agents（搜索/代码/分析）→ 
各 Agent 执行并返回结果 → Master Agent 整合回答
```

### 3.3 场景三：跨服务 Agent 调用

```
本地 Agent 收到请求 → 发现远程 Agent 能力 → 
发送 A2A 任务 → 远程 Agent 执行 → 返回结果 → 
本地 Agent 整合回答
```

### 3.4 场景四：运维治理

```
管理员配置规则 → Agent 请求经过 Harness → 
检查护栏/规则/预算 → 执行 LLM → 检查输出护栏 → 
记录成本/质量 → 检测回归 → 自动回滚（如需要）
```

## 4. 核心概念

### 4.1 Agent

Agent 是具有工具调用能力的 AI 实体：
- **工具**：搜索、代码执行、天气查询等
- **MCP 协议**：标准化工具接口
- **多 Agent**：协作完成任务

### 4.2 知识库（Knowledge）

知识库是文档的向量存储：
- **文档**：原始文件（PDF/Word 等）
- **Chunk**：文档分块后的片段
- **Embedding**：Chunk 的向量表示
- **检索**：基于相似度找相关 Chunk

### 4.3 记忆（Memory）

记忆是跨会话的信息持久化：
- **短期记忆**：当前会话的消息历史
- **长期记忆**：重要信息存入向量库
- **召回**：基于 query 找相关记忆

### 4.4 A2A（Agent-to-Agent）

A2A 是跨服务的 Agent 通信协议：
- **AgentCard**：Agent 元数据（能力、URL）
- **Task**：任务实体（状态、消息）
- **生命周期**：Submitted → Working → Completed/Failed

### 4.5 Harness

Harness 是运维治理系统：
- **Gate**：请求经过的多层检查点
- **护栏**：内容过滤（注入检测、敏感信息）
- **规则**：约束检查（Token 限制、工具权限）
- **预算**：成本控制（租户配额）
- **评估**：质量评分（Faithfulness、Relevancy）
- **A/B 测试**：模型/提示词对比实验

## 5. 技术决策

### 5.1 为什么选择微服务

| 决策 | 原因 |
|------|------|
| 服务拆分 | Chat/Knowledge/Memory 等职责清晰 |
| 独立扩展 | Knowledge 可能需要更多资源 |
| 故障隔离 | 单服务故障不影响其他 |
| 技术演进 | 可独立升级各服务 |

### 5.2 为什么选择 gRPC

| 决策 | 原因 |
|------|------|
| 性能 | Protobuf 二进制序列化，比 JSON 快 |
| 类型安全 | 强类型接口定义 |
| 流式支持 | 支持双向流（实时对话） |
| 跨语言 | 未来可支持其他语言客户端 |

### 5.3 为什么选择 Go

| 决策 | 原因 |
|------|------|
| 性能 | 高性能，适合网络服务 |
| 并发 | Goroutine 轻量级并发 |
| 生态 | gRPC/Docker/K8s 生态完善 |
| 维护 | 相比 Rust 更易维护 |

## 6. 项目边界

### 6.1 包含的功能

✅ Agent 对话 + 工具调用
✅ 多 Agent 协作
✅ A2A 跨服务通信
✅ MCP 工具协议
✅ 知识库（上传、分块、检索）
✅ 长期记忆
✅ Harness 运维治理（全部功能）

### 6.2 不包含的功能

❌ LangGraph 状态机（复杂度高，简化为 Pipeline）
❌ 7 种对话压缩策略（仅保留滑动窗口 + 摘要）
❌ HyDE/MultiQuery 等 RAG 增强（基础检索足够）
❌ 复杂三层内存（简化为两层：短期 + 长期）

| API P99 延迟 | < 500ms |
| 检索召回率 | > 80% |
| 评估通过率 | > 90% |
| A/B 测试周期 | < 24h 判断显著性 |
